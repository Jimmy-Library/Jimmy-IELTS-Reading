# 04 · 同题型 · 不同操作逻辑(差异分析)

> 同一个 `kind` 值,实际写法和操作体验可能差异巨大。本文整理已观察到的 5 类差异,并给出兼容性建议。Developed by **Jimmy**

---

## 总览

| 表面 kind | 差异点 | 体验差异 | 受影响题数(粗估) |
| -------- | ------ | -------- | ------------------ |
| `matching` | dropzone 位置:文中 vs 列表中 | 拖拽距离 / 滚动负担 | 89 题中两种各占约一半 |
| `matching` | 是否允许选项重用(`allowOptionReuse`) | 选项池行为 | 与 `classification` 之间界限模糊 |
| `summary_completion` | 输入方式:文本框 vs 拖词到 span | 是否需要打字 | 82 题里两种皆有 |
| `true_false_not_given` ↔ `yes_no_not_given` | 字面 TRUE / YES 不同 | 用户必须严格选对 vs 自动归一 | 全部 119 题受归一逻辑影响 |
| `short_answer` ↔ `sentence_completion` | 是否带题干句模板 | 阅读体验完全不同 | 39 + 69 题 |
| `multi_choice` | name 共用 vs 题号独立 | 多选一组 vs N 个独立单选 | 34 组 |

---

## 1. matching — DOM 位置的两种写法

同样在 schema 里都是 `"kind": "matching"`,但 `bodyHtml` 实际产出两种很不同的页面。

### 写法 A:Dropzone 与段落并列(heading matching 典型)

```html
<div class="paragraph-wrapper" id="q1-anchor">
    <div class="paragraph-dropzone dropzone" data-question="q1">
        <span class="paragraph-label">Paragraph A (Q1):</span>
        <div class="dropped-items"></div>
    </div>
    <p><strong>A</strong> 占位段落正文 …</p>
</div>
```

- **DOM**:`dropzone` 紧贴在段落上方
- **用户视线**:边读段落边把 heading 拖到上方框
- **操作距离**:从底部 pool 拖到上方,**滚动距离可能大**
- **失败模式**:不小心放进 `.dropped-items` 内部时,`getDropzoneAnswer` 仍然能识别(查找 `.drag-item`)

### 写法 B:Dropzone 与题号并列(statement matching 典型)

```html
<div class="match-question-item" id="q9-anchor">
    <div class="match-dropzone" data-question="q9"></div>
    <p><strong>9</strong> 占位陈述</p>
</div>
```

- **DOM**:`match-dropzone` 直接在题号旁
- **用户视线**:固定向下扫问题列表
- **操作距离**:短(pool 紧贴问题列表下方)
- **失败模式**:`.match-dropzone` 与 `.paragraph-dropzone` 选择器不同,`unifiedReadingPage.getDropzoneAnswer` **两种都查**,所以兼容

### 兼容性边界

[unifiedReadingPage.js](../js/runtime/unifiedReadingPage.js) `getDropzoneAnswer(qN)` 内部的选择器优先级:

```js
[
    `.match-dropzone[data-question="${qN}"]`,
    `.match-dropzone[data-question-id="${qN}"]`,
    `.drop-target-summary[data-question="${qN}"]`,
    `.drop-target-summary[data-question-id="${qN}"]`,
    `.dropzone[data-question="${qN}"]`,
    `.paragraph-dropzone[data-question="${qN}"]`
]
```

→ **不要写 `.drop-zone`(连字符)或 `.dropZone`(驼峰)**,会被 `getDropzoneAnswer` 完全错过。

### 编辑器里如何处理

[tools/reading-editor.html](tools/reading-editor.html) 提供两个变体:

- `matching · heading→paragraph`(写法 A)
- `matching · statement→option`(写法 B)

两者最终 `kind` 都是 `matching`,差异在 HTML 模板。

## 2. matching ↔ classification 边界

### 现象

- **Schema 上**:`matching` 默认 `allowOptionReuse: false`,`classification` 默认 `allowOptionReuse: true`
- **HTML 上**:两者一模一样(都是题号 + dropzone + 选项 pool)
- **评分上**:完全一致

### 内部差异

唯一区别:**前端拖拽逻辑**会读 `allowOptionReuse` 判断"拖出后选项池是否还保留这个 chip"。

```js
// 简化逻辑
function onDrop(item, zone) {
    placeIntoZone(item, zone);
    if (!groupAllowsReuse) {
        removeFromPool(item);              // matching:消耗选项
    }
}
```

### 已知混乱

题库里仅有 **1 题** 显式声明 `kind = "classification"`,大量本应是分类的题目仍写成 `matching`。这是历史遗留。

### 维护建议

- 新建题目时:
  - 选项**会重复**(同一选项分给多个题)→ 用 `classification`
  - 选项**不可重复**(消耗式)→ 用 `matching`
- 即使继承的题目里 kind 写错,也不要批量改 → 因为 `allowOptionReuse` 才是真正的开关,只要这字段对就行

## 3. summary_completion — 文本输入 vs 拖词

同样是 `summary_completion`,但用户输入方式有两种。

### 输入式(无词库)

```html
<p>占位段落:咖啡的发源地是
   <input type="text" data-question="q27" class="answer-input">
   ,首次商业种植开始于
   <input type="text" data-question="q28" class="answer-input">.
</p>
```

- **采集**:`getTextualAnswer(qN)` → `input.value.trim()`
- **失败模式**:大小写 / 标点 / 复数差异 → `AnswerMatchCore` 已经包容大部分

### 拖词式(有词库)

```html
<div class="word-bank">
    <strong>List of Words</strong>
    <div class="pool-items">
        <div class="drag-item" draggable="true" data-word="origin">origin</div>
        <div class="drag-item" draggable="true" data-word="cultivation">cultivation</div>
    </div>
</div>

<p>占位段落:咖啡的发源地是
   <span class="drop-target-summary dropzone" data-question="q27"></span>,
   首次商业种植开始于
   <span class="drop-target-summary dropzone" data-question="q28"></span>。
</p>
```

- **采集**:`getDropzoneAnswer(qN)` → 读拖入 `.drag-item` 的 `data-word`
- **失败模式**:用户输入时**没有词库提示**,但题目实际要求只能从词库选 → 这时输入式是错的设计

### 编辑器如何区分

编辑器在 `summary_completion` 表单里有一个**复选框**:

> ☐ **附带词库(Word Bank)** ── 勾选后会显示词库编辑器

勾选时:输出 `<span class="drop-target-summary dropzone">`,自动生成 `.drag-item` 词库
未勾选:输出 `<input class="answer-input">`,纯输入

## 4. TFNG vs YNNG — 用户感知的最大差异题型

| 维度 | `true_false_not_given` | `yes_no_not_given` |
| ---- | ---------------------- | ------------------ |
| 用途 | **事实**判断 | **观点**判断 |
| 选项字面 | TRUE / FALSE / NOT GIVEN | YES / NO / NOT GIVEN |
| 答案 Key 字面 | `"TRUE"` | `"YES"` |
| 用户必须选对的字面? | 表面看是 | 表面看是 |
| 实际是否区分? | **否**(归一化层都变 `true`/`false`) | **否** |

### 隐含坑

由于 `AnswerMatchCore.normalizeToken('YES') === 'true'`:

- 用户在 TFNG 题目里写 `YES` → 系统判它等价于 `TRUE` → **得分**
- 用户在 YNNG 题目里写 `TRUE` → 也得分

这在练习场景是友好的,但在**严格考试模拟**场景下应该报错。

### 修复方法(如果你想要严格模式)

在 [js/utils/answerMatchCore.js](../js/utils/answerMatchCore.js) `BOOLEAN_SYNONYMS` 之外加一个**严格模式**:

```js
const STRICT_MODE = false;   // 暴露成可切换的配置

const BOOLEAN_SYNONYMS = STRICT_MODE
    ? new Map([['true','true'], ['false','false']])              // 严格,只接受字面
    : new Map([['true','true'], ['t','true'], ['yes','true'], ['y','true'],
               ['false','false'], ['f','false'], ['no','false'], ['n','false']]);
```

然后在题组级别加 `strictAnswers: true` 标志,渲染时把全局开关临时切换。

→ 不建议默认开启,会让 90% 的用户体验变差。

## 5. short_answer vs sentence_completion — 容易混淆

### 字面体验

| 维度 | `short_answer` | `sentence_completion` |
| ---- | -------------- | --------------------- |
| 题面 | 完整问句 | 半句 + 空格 |
| 输入框位置 | 题面后 | 句子中间 |
| 字数提示 | 通常 1–3 词 | 通常 1–3 词 |
| 答案 Key | 与 sentence_completion 完全一致 | 同上 |
| 评分 | 完全一致 | 同上 |

### 例子对比

#### short_answer

```html
<p><strong>1</strong> 占位问题:风车的核心动力部件是什么?</p>
<input type="text" data-question="q1" class="answer-input">
```

#### sentence_completion

```html
<p><strong>1</strong> 风车的核心动力部件是
   <input type="text" data-question="q1" class="answer-input">。
</p>
```

→ 后端逻辑**完全等价**,只是排版思路不同。

### 维护建议

- 不要为了"换种 kind"批量改 → 用户题型分布平衡更有价值
- 编辑器允许在生成时**互转**:同样的题干写两遍排版即可

## 6. multi_choice — name 共用 vs 题号独立

### 写法 A:N 个独立 radio(伪多选)

```html
<div class="question-item" id="q18-anchor">
    <p><strong>18</strong> 第一个选?</p>
    <ul class="choice-list">
        <li><input type="radio" name="q18" value="A">A</li>
        <!-- ... -->
    </ul>
</div>
<div class="question-item" id="q19-anchor">
    <p><strong>19</strong> 第二个选?</p>
    <ul class="choice-list">
        <li><input type="radio" name="q19" value="A">A</li>
        <!-- ... -->
    </ul>
</div>
```

- **用户体验**:必须给每题选一个,等于两个独立单选
- **评分**:`q18` 与 `q19` 分别比对

### 写法 B:checkbox 共享 name(真多选)

```html
<p>Choose <strong>TWO</strong> letters, A–E.</p>
<ul class="choice-list">
    <li><input type="checkbox" name="q18-19" value="A">A</li>
    <li><input type="checkbox" name="q18-19" value="B">B</li>
    <li><input type="checkbox" name="q18-19" value="C">C</li>
    <li><input type="checkbox" name="q18-19" value="D">D</li>
    <li><input type="checkbox" name="q18-19" value="E">E</li>
</ul>
```

- **用户体验**:在同一个列表里勾选 2 项
- **评分**:`collectAnswers` 内 `getCheckboxAnswers` 把 `name="q18-19"` 的勾选集合按 ASCII 排序后,1:1 拆给 `[q18, q19]`

### 已知坑

- **题号合并必须连号**:`q18-19` ✓,`q18-20` ✓(代表 q18/q19/q20),`q18-q19` ✗
- 用户只勾 1 个会得 1 分(`q18 = 'A', q19 = ''`)
- 用户勾 3 个时,`q18 = 'A', q19 = 'B'`,**第三个被丢弃**

### 编辑器如何处理

在 `multi_choice` 表单里:

- 默认生成**写法 B**(更接近真实考试)
- 提供"展开为独立单选"按钮 → 一键转 A 写法

## 7. 整体兼容性建议

### 建议遵守

1. **优先用 `data-question="qN"`**,不要用 `id="qN"`(易冲突)
2. **rado/checkbox 用 `name` 而不是 `data-question`**(浏览器原生分组机制更可靠)
3. **拖拽源**:统一用 `class="drag-item"`,数据字段统一用以下三选一:
   - `data-heading="iv"`(标题匹配)
   - `data-option="A"`(选项匹配 / 分类)
   - `data-word="harvest"`(词库)
4. **拖拽目标**:`class="dropzone"` 是基类,其它派生类(`paragraph-dropzone` / `match-dropzone` / `drop-target-summary`)都自动兼容

### 应避免

1. ❌ 用 `<button>` 当拖拽源 → 浏览器拖拽行为不一致
2. ❌ 在 `bodyHtml` 内写 `<script>` → 会被沙箱屏蔽
3. ❌ 用 inline `onclick="…"` → 与 unifiedReadingPage 的事件委托冲突
4. ❌ 在 `data-question` 写中文 / 空格 → 选择器无法 escape

## 8. 排查清单(发现"题型显示怪异"时)

1. **答案永远空** → 控件没挂 `data-question` / `name`
2. **拖了没反应** → dropzone class 拼错(应为 `dropzone` / `match-dropzone` / `paragraph-dropzone`)
3. **选项池消耗了不该消耗** → `allowOptionReuse` 设置反了
4. **多选只算 1 个** → checkbox 的 `name` 没用连号合并
5. **TFNG 用户写小写没分** → 不会发生,归一层包容(若发生,检查是不是用了 `Y/N`)
6. **图示题完全不显示图片** → 路径不对,改成相对路径并放进 `assets/`
7. **导出 PDF 显示乱码** → 答题页含未声明编码的字符,在 `<head>` 加 `<meta charset="utf-8">`

---

## 总结

| 你看到这种现象 | 应该想到 |
| ---------- | -------- |
| 同 kind 不同布局 | bodyHtml 里 DOM 写法不一致 |
| matching ↔ classification 几乎一样 | 仅 `allowOptionReuse` 不同 |
| TFNG 与 YNNG 互通 | `AnswerMatchCore` 归一化 |
| 多选拆题号 vs 独立题号 | name 共用规则 |
| 提交后某题永远空 | 控件缺 `data-question` |

→ **本系统的核心设计**:**kind 是元信息,行为由 DOM 属性决定**。这意味着新增题型大多数情况下不需要动核心代码,但也意味着 bodyHtml 写错时排查会迷茫,记得对照本表。
